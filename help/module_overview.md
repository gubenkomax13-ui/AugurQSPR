# Карта модулей Augur QSPR

Этот файл описывает модули не с точки зрения файлов Python, а с точки зрения пользователя: зачем нужен раздел, какие инструменты внутри и какую задачу они закрывают.

## Данные и дескрипторы

**Цель:** собрать корректную матрицу признаков для QSPR-модели.

**Инструменты:**
- загрузка CSV/XLSX с SMILES и целевым свойством;
- расчёт RDKit, Mordred, PaDEL, спектральных, xTB, DScribe и пользовательских дескрипторов;
- очистка числовой матрицы: пропуски, бесконечные значения, константные признаки;
- выбор источника дескрипторов: молекулярные, спектральные, пользовательские или объединённые.

**Чем отличаются инструменты:** загрузка отвечает за исходные данные, расчёт дескрипторов превращает структуры в признаки, очистка делает матрицу пригодной для ML, а выбор источника задаёт химическое пространство модели.

**Соответствие цели:** хорошее. Модуль закрывает путь от таблицы веществ до числовой матрицы. Важно явно показывать пользователю, какие строки/дескрипторы были отброшены.

## Обучение модели

**Цель:** обучить регрессионную модель для заданного свойства.

**Инструменты:**
- выбор алгоритма;
- настройка гиперпараметров;
- автоматический отбор дескрипторов;
- обучение финальной модели;
- сохранение выбранной модели.

**Чем отличаются инструменты:** алгоритм задаёт тип зависимости, гиперпараметры управляют сложностью, отбор дескрипторов снижает шум и риск переобучения, сохранение фиксирует проверенную модель.

**Соответствие цели:** хорошее. Следующий полезный шаг — сильнее связать обучение с расширенной валидацией, чтобы модель не считалась готовой только по train/CV метрикам.

## Базовая валидация

**Цель:** быстро оценить качество модели на данных, не использованных для конкретного прогноза.

**Инструменты:**
- Hold-out / random split;
- K-Fold;
- Leave-One-Out;
- External Validation Simulator.

**Чем отличаются инструменты:** Hold-out даёт один независимый test, K-Fold усредняет качество по нескольким фолдам, LOO подходит для очень малых наборов, External Validation Simulator проверяет удалённые объекты.

**Соответствие цели:** хорошее для базовой оценки. Для оценки переносимости нужен расширенный модуль.

## Расширенная валидация и переносимость

**Цель:** понять, переносится ли модель на новые химические области, а не только хорошо работает на случайных разбиениях.

**Инструменты:**
- Repeated K-Fold;
- Group / Scaffold split;
- Learning curves;
- Prediction interval hold-out coverage;
- планируемый Split Strategy Comparison.

**Чем отличаются инструменты:** Repeated K-Fold проверяет устойчивость метрик, scaffold/group split проверяет перенос между семействами, learning curves показывают дефицит данных и переобучение, coverage audit проверяет честность интервалов неопределённости.

**Соответствие цели:** частично завершено. Уже есть основные инструменты, но нужен единый сравнительный запуск random/scaffold/cluster/group/external-distance split.

## Диагностика модели

**Цель:** объяснить, где модель ошибается и какие признаки использует.

**Инструменты:**
- Applicability Domain;
- Williams plot;
- residual diagnostics;
- важность дескрипторов: коэффициенты, native importance, permutation importance, SHAP, unified importance.

**Чем отличаются инструменты:** AD отвечает на вопрос “можно ли доверять прогнозу для этого объекта”, residual diagnostics ищет систематические ошибки, importance объясняет вклад дескрипторов.

**Соответствие цели:** хорошее. Для полной ясности unified importance стоит привести к единому формату descriptor/method/importance/direction/stability/warning.

## Анализ ошибок

**Цель:** найти химические причины ошибок модели.

**Инструменты:**
- структурные аннотации;
- scaffold/family/series summaries;
- анализ систематических пере- и недооценок;
- проверка ошибок по структурным группам.

**Чем отличаются инструменты:** таблицы ошибок показывают отдельные проблемные вещества, группировка по семействам показывает, где ошибка становится системной.

**Соответствие цели:** хорошее. Модуль логично дополняет scaffold/group validation: один показывает падение качества, другой помогает понять химическую причину.

## Область применимости и неопределённость

**Цель:** оценить надёжность прогноза для нового вещества.

**Инструменты:**
- distance AD;
- descriptor range AD;
- leverage AD;
- structural similarity AD;
- conformal prediction interval;
- bootstrap, consensus, local analogue error, GPR uncertainty;
- reliability score.

**Чем отличаются инструменты:** AD оценивает похожесть объекта на train, prediction interval даёт диапазон ожидаемого значения, bootstrap/consensus/local/GPR показывают разные источники неопределённости.

**Соответствие цели:** хорошее. Важно не смешивать prediction interval и диагностические std: interval отвечает за покрытие, std — за разброс или локальную нестабильность.

## Сравнение моделей

**Цель:** выбрать модель не только по одной метрике, а по совокупности качества, устойчивости и применимости.

**Инструменты:**
- таблица моделей;
- сравнение train/CV/hold-out/Repeated Hold-out/Bootstrap/Y-randomization;
- рейтинги и предупреждения.

**Чем отличаются инструменты:** метрики качества показывают точность, проверки устойчивости показывают риск случайного результата, предупреждения помогают не выбрать красивую, но ненадёжную модель.

**Соответствие цели:** хорошее, но стоит добавить результаты расширенной валидации и split strategy comparison в общий рейтинг.

## Прогноз новых веществ

**Цель:** применить проверенную модель к новым SMILES или таблице дескрипторов.

**Инструменты:**
- расчёт/загрузка дескрипторов для новых объектов;
- прогноз свойства;
- uncertainty analysis;
- AD-сигналы;
- таблица надёжности прогноза.

**Чем отличаются инструменты:** прогноз даёт значение, uncertainty и AD объясняют, насколько ему можно доверять.

**Соответствие цели:** хорошее. Лучший сценарий — показывать прогноз вместе с причинами низкой надёжности, а не только число.

## Отчёт

**Цель:** собрать результаты в понятный документ.

**Инструменты:**
- методология;
- таблицы метрик;
- графики;
- коэффициенты/важности;
- выводы и ограничения.

**Чем отличаются инструменты:** методология описывает как модель построена, метрики показывают качество, графики и ограничения помогают читателю понять границы применимости.

**Соответствие цели:** хорошее. Следует включить новые advanced validation результаты в отчёт после стабилизации формата таблиц.

## SAOD и структурная проверка

**Цель:** проверить структурную достоверность и химические паттерны, которые могут влиять на интерпретацию модели.

**Инструменты:**
- structural filters;
- SMARTS/pattern rules;
- SAOD checks;
- анализ серий и подозрительных структурных случаев.

**Чем отличаются инструменты:** фильтры проверяют формальные условия, SMARTS ищет заданные фрагменты, SAOD помогает обнаружить химически подозрительные или особые случаи.

**Соответствие цели:** хорошее. Модуль полезен до обучения и после анализа ошибок.

## Итоговая логика

Минимальный путь пользователя:

1. Подготовить данные и дескрипторы.
2. Обучить несколько моделей.
3. Запустить базовую валидацию.
4. Запустить расширенную проверку переносимости.
5. Проверить AD, ошибки и важность дескрипторов.
6. Выбрать модель по качеству, устойчивости и честности uncertainty.
7. Сохранить модель и сформировать отчёт.
